## 0. デザイン原則（このゲーム固有の制約）

1. **色は意味を1人で背負わない（三重冗長）**: カード種別は必ず「色＋アイコン＋ラベル」の3つで冗長表現する。色覚多様性対応であると同時に、これ自体が EventStorming の学習効果（記法の刷り込み）になる。色だけ／アイコンだけ／ラベルだけの表現は禁止。アイコンは lucide SVG（`Icon.tsx`）で統一する。
2. **教科書準拠カラーを死守**: EventStorming 業界標準色をズラさない。学んだ人が実ワークショップで混乱しないことが最優先。ブランド演出は**アクセント色（ティール）と温かい地（クラフト紙／ダーク墨）と報酬の金**で行い、カード7色には触れない。アクセントはカード7色と色相が衝突しないティールを選ぶ。
3. **付箋メタファに忠実**: カードは「ホワイトボードに貼った付箋」。角丸は小さく、わずかに回転し、物理的な影を落とす。角丸が大きい・フラットすぎると付箋に見えない。
4. **情報的フィードバック中心**: 報酬演出は「達成したという情報」を伝えるためにある。ポイント加算の派手さで快楽報酬を作らない（過正当化効果の回避）。「なぜ正解／不正解か」を必ず言葉で返す。
5. **大きめフォント・高コントラスト**: 独学者が集中して読む教材。本文は最低 13px、できれば14–16px。日本語の可読性を最優先する。

### 2.1 画面遷移図

- **凡例（legend）**: 6〜7枚の付箋チップを並べる。プロトは `aria-hidden` だったが、本仕様では**読み上げ対象にする**（学習者にとって用語一覧は重要情報）。各チップは「アイコン＋ラベル」を必ず表示（三重冗長）。横に溢れたら横スクロール（モバイル）。
- **ステージノード**: 縦のタイムライン（破線の道）上にノードを並べる。状態は `done(✅・緑) / current(⚡・オレンジ・鼓動) / locked(🔒・グレー)` の3つ。アイコン＋テキスト＋星で状態を冗長表現（色のみに依存しない）。
- locked ノードは `disabled` + `aria-label="（ロック中）"`。フォーカス可だが活性化しない。

- MODE2 は「正しいトリガーのみ着地可、誤りは弾く」。全接続で `Phase8 不変条件 = 充足` → 自動でクリア演出。
- **不変条件チェッカー**は MODE2 で常時表示。「孤立コマンド（起動経路の宙吊り）をゼロに」を可視化する学習装置（CONTEXT.md の Orphan Command）。`✓/✗` は色だけでなく記号＋テキストで示す。

## 3. ゲーミフィケーション

> 設計の北極星は **完走率とシェア率**。リテンション（毎日来させる）は追わない（DESIGN.md §6）。
> 全メカニクスは「情報的フィードバック」として設計し、報酬の獲得自体を目的化させない（過正当化効果の回避）。

### 3.1 星評価（★初クリア / ★★ヒント不使用 / ★★★ノーミス）

- **評価軸を毎回明示する**（リザルトに凡例を出す）。「なぜ★2なのか」が分かることが学習価値。隠れた減点は作らない。
- 判定は純粋関数（`src/game/` 側）。デザイン要件は**表示の透明性**:
  - `★` = 初クリア（必ず1つは付く＝肯定から入る）。
  - `★★` = それに加えヒント未使用。
  - `★★★` = さらにノーミス（誤配置0・順序判定リトライ0）。
- 星は**情報**であり快楽報酬の演出を盛らない。獲得時は短い「カチッ」音とわずかな拡大（150ms）のみ。コインシャワー等の射幸演出はしない。
- **再挑戦動機**: ベストスコア保持（`Math.max`）。★3未満のステージはマップで「★★☆」と空き星を見せ、間隔反復（再挑戦で習熟）を促す。これが市場調査の「3つ星＝間隔反復のゲーム化」。

### 3.2 用語図鑑（コレクション）

> 市場調査「1人完結型教材ではストリークより**コレクション欲求**が有効」。本ゲームの主要エンゲージメント装置。

```
┌─ 用語図鑑 ──────────────────────────────┐
│ ⚡ イベント   ▶ コマンド   👤 アクター     │ ← 獲得済(色つき・表)
│ ⚙ ポリシー   ☁ 外部システム 📋 リードモデル │
│ ┌──────┐                                 │
│ │ ?    │ ← 未獲得(シルエット・空きスロット)  │
│ │ 集約  │   "第2章で解放"                  │
│ └──────┘                                 │
└─────────────────────────────────────────┘
```

- 各ステージクリアで `vocab`（stages.ts の `vocab`）を1枚獲得 → 図鑑が1マス埋まる。
- **空きスロットを必ず見せる**（学習動機。CONTEXT.md「空きスロットが学習動機になる」）。未獲得は `?` シルエット＋ラベルのみ＋「第2章で解放」。
- 図鑑カードを開くと: JA／EN／定義／標準色スウォッチ／アイコン／時制ルール（イベント=過去形 等）を表示。これ自体が**記法リファレンス**として機能。
- カード詳細は色＋アイコン＋ラベル＋定義の4点で必ず冗長表現（CVD配慮）。
- **開封率は学習意図の代理指標**（DESIGN.md 成功指標）。図鑑タップ／カード開封をログする。

### 3.3 完走時のシェア用サマリ

- 目的: **シェア率を上げる**（北極星指標）。SNS（X / はてブ / Zenn）で語られる状態を作る。
- 生成物: テキスト＋（任意で）OG画像風のカード。例:
  ```
  StormQuest 第1章「ピザデリバリー」クリア ⚡▶👤⚙☁📋
  獲得 ★10/12 ・ ノーミス3ステージ ・ 用語6/7習得
  EventStormingの"色と文法"を15分で。 #StormQuest
  ```
- **Web Share API**（`navigator.share`）を第一手段、未対応はクリップボードコピー＋X 投稿リンクにフォールバック。
- カードに7種の付箋アイコン列（⚡▶👤⚙☁📋📦）を入れ、**EventStorming記法が一目で伝わる**ビジュアルにする（=ツール自体の説明になる）。
- 誇張しない: 「記法入門を完走」であり「DDDマスター」とは書かない（DESIGN.md 正直な境界）。

### 3.4 ヒント（段階フェードアウト）

> Flexbox Froggy 批判「ヒント過多」を回避。市場調査「ヒントを段階的にフェードアウト」。

- MODE1 ヒントは3段階を**手動開放**（自動で答えを見せない）:
  1. **L1（弱）**: シナリオ内の時系列キーワードをハイライト（「注文して」「支払いを済ませ」…）。
  2. **L2（中）**: 最初の1枚の正しい位置だけ示す（残りは自力）。
  3. **L3（強）**: 隣接1ペアの前後関係を1つ提示。
- ヒントを使うと★★（ヒント不使用）を逃す。この**トレードオフを開放前に明示**（「ヒントを使うと★1つ減ります」）。情報的に納得して使わせる。
- 後のステージほどヒントの初期可視性を下げる（フェードアウト＝足場外し）。

### 3.5 ストリーク（連続日数）を入れない理由 — 明記

- **DESIGN.md §5「作らないもの」にデイリーストリーク・通知が明記**されている。本デザインもこれに従う。
- 根拠1: **北極星が完走率／シェア率でリテンションを追わない**。ストリークは毎日の再訪を促す装置で、本プロダクトの指標と方向が違う。
- 根拠2: **過正当化効果**。「連続を切らさない」という外的圧力は、学習そのものの内発的動機（記法が腑に落ちる楽しさ）を侵食する。市場調査の「報酬は達成の情報的フィードバックとして設計する」に反する。
- 根拠3: **非同期・即プレイの設計思想**。「いつでも来て15分で1概念」が価値であり、来訪を義務化すると体験が損なわれる。
- 根拠4: **1人完結型教材ではコレクション欲求のほうが有効**（市場調査）。エンゲージメントは図鑑と★3再挑戦で作る。
- 代替: 進捗は「章の完成度（マップの埋まり）」「図鑑の充足」で内的に可視化する。プレッシャーでなく達成で引く。

## 4. 付箋メタファのビジュアル

### 4.1 付箋の見た目

```css
.sticky {
  background: var(--card-bg);          /* CARD_META[kind].color を CSS変数注入 */
  color: var(--card-ink);              /* CARD_META[kind].ink */
  border-radius: var(--radius-sticky); /* 3px = 紙の角 */
  box-shadow: var(--shadow-sticky);    /* 1px 3px 6px rgba(0,0,0,.22) 物理影 */
  font-size: var(--fs-sticky);
  font-weight: 700;
  line-height: var(--lh-tight);
  display: flex; flex-direction: column; align-items: center; justify-content: center;
  text-align: center; gap: 2px;
  padding: var(--space-2xs) var(--space-xs);
  min-width: 88px; min-height: 64px;   /* タップターゲット ≥44px を内包。プロトより一回り大きく */
  user-select: none; -webkit-user-select: none;
  touch-action: none;                  /* dnd-kit のため */
}
/* アクターのみ「黄(小)」= EventStorming 記法どおり一回り小さく */
.sticky--actor { min-width: 72px; min-height: 56px; font-size: var(--fs-sticky-sm); }
```

- **三重冗長の内部構造**: 付箋の中は `[アイコン] + [ラベル(主) ] + [種別名(小)]` を縦に積む。例: `⚡ / 注文が確定した / イベント`。種別名は `--fs-caption` の極小だが必ず入れる（色が見えない人の生命線）。
- **質感（紙っぽさ）**: わずかな内側ハイライト `inset 0 1px 0 rgba(255,255,255,.4)` を足し、フラットすぎないようにする。テクスチャ画像は使わない（軽量・CLS回避）。
- **回転**: 配置済み・凡例・図鑑では `transform: rotate(±1.5deg)` を交互に付け、ホワイトボードに人が貼った"手作業感"を出す。手札では回転0（読みやすさ優先）。`nth-child(even/odd)` で符号を変える。

### 4.2 配置時の juice（やりすぎない）

> juice は優先順位③。②学習の誠実さ・①正確さを侵さない範囲で。「気持ちいいが、学びを邪魔しない」。

| タイミング | 演出 | 値 |
|---|---|---|
| 掴む（drag start） | わずか拡大＋回転＋影が伸びる | `scale(1.08) rotate(-4deg)` / `--shadow-sticky-drag` |
| ドロップ可能スロット上 | スロットが `--accent-tint` に発光・点線→実線寄り | `border-color: --accent; background: --accent-tint` |
| 着地（スナップ） | スロットへ吸着＋軽い行き過ぎ戻り | `--ease-out` で 200ms、`overshoot` 4% |
| 正解で着地 | スロットが一瞬緑パルス（1回） | `--state-ok` の outline が 450ms で拡散消失 |
| MODE2 接続成立 | ワイヤが赤破線→緑実線、`?`→`✓` | 250ms |
| 全クリア瞬間 | 盤全体に薄い祝福フラッシュ1回（控えめ） | opacity フェード、コンフェッティは**最小限**（完走画面のみ可） |

- **効果音の使いどころ**（すべて任意・OFFでも完全にプレイ可能 §6）:
  - スナップ着地: 「コトッ」（短い・低め）
  - 正解接続: 「ポン↑」（上昇音・肯定）
  - 誤配置で弾く: 「コッ」（短い否定・不快にしない・音量控えめ）
  - 星獲得: 「カチッ×星数」
  - 完走: 短いファンファーレ（2秒以内）
  - 音は WebAudio で軽量生成 or 小さい mp3。**初期設定 OFF も検討**（公共の場でのプレイ配慮）。設定でいつでも切替。

### 4.3 正解 / 誤配置のフィードバック表現

- **誤配置（弾く）**: カードがスロットに入らずシェイク（`--motion-emphasis`）＋赤枠一瞬。**トーストで理由を必ず提示**。理由文は cardMeta の種別名を使い「それはコマンド（水色 ▶）。イベントは過去に起きた事実です」のように**正しい記法へ誘導**する（stages.ts の `distractor.reason` をそのまま使用）。これが情報的フィードバックの核。
- **順序違反（MODE1 こたえあわせ）**: 間違った位置のカードのみ赤枠。「順序が違うカードが N 枚（赤枠）。タップで手札に戻して入れ替えよう」。**破壊しない**＝置いたものは消えず、何度でも直せる（失敗の心理的コストを下げる）。
- **正解**: 緑パルス＋肯定音＋（MODE2なら）ワイヤ接続演出。**「なぜ正解か」を急かさずクリア後の reality で回収**（その場で説教しない）。
- 色だけに頼らない: 誤=赤枠＋シェイク＋`✗`＋トースト文、正=緑枠＋`✓`＋音＋テキスト。視覚・モーション・テキスト・（任意で）音の複数経路。

## 5. モバイル D&D 体験

### 5.1 レイアウト方針

| | PC（≥720px） | スマホ（<720px） |
|---|---|---|
| MODE1 タイムライン | 横並び（時間軸が左→右に一直線）。スロットを `flex` 横一列、溢れたら2段折返し | **縦スクロール＋スナップ**。スロットを縦に積み、時間は上→下。`scroll-snap-type: y proximity` |
| MODE2 トリガー | 横: `[トリガー]══[コマンド]` の対を縦に積む | 同左。対は横のまま、対同士を縦に積む |
| 手札／トレイ | 盤の下に横スクロールトレイ | 画面下固定の横スクロールトレイ（親指が届く位置） |

- スマホのタイムラインは「時間 = 縦方向」に倒す案も検討したが、**矢印ラベルで時間方向を明示**すれば横折返しでも可。実装は「横並び flex-wrap、各行に番号、末尾に `時間 →`」を基本とし、狭幅では2列に落とす。`scroll-snap` は手札トレイと縦長盤の両方に適用し、カードが中途半端に切れないようにする。
- **手札は常に親指圏（画面下 1/3）に固定**。盤は上、操作は下、の分離を守る。

### 5.2 D&D 実装方針（dnd-kit）

- `@dnd-kit/core` を使用。**TouchSensor・PointerSensor・KeyboardSensor を初日から全部オン**（DESIGN.md / CLAUDE.md 必須要件）。HTML5 ネイティブ DnD は使わない（モバイル不可）。
- `TouchSensor` は `activationConstraint: { delay: 120, tolerance: 8 }`（誤ドラッグ防止・スクロールと両立）。
- `touch-action: none` を付箋に付け、ドラッグ中のページスクロールを抑止。
- ドラッグ中は `DragOverlay` で実体を持ち上げ、元位置にはゴースト（薄い枠）を残す。
- スナップ: `closestCenter` 衝突判定。ドロップ後 200ms `--ease-out` で着地アニメ。

### 5.3 タップ操作の代替動線（必須）

> D&D が苦手／できない環境（スクリーンリーダー・運動制約・小画面）のため、**ドラッグなしで完全クリア可能**にする。

- **2タップ配置**: カードをタップ → 選択状態（拡大＋枠）＋「置きたいスロットをタップ」トースト → スロットタップで配置。
- 配置済みカード（MODE1）はタップで手札に戻る。
- 選択中の再タップで選択解除。
- この代替は**プロト実装済みの挙動を踏襲**（`handleCardTap` / `onSlotTap`）。dnd-kit と共存させる（センサーの activation 前のタップを拾う）。
- 手札ラベルに操作説明を常時表示: 「ドラッグ、またはタップで選択 → スロットをタップ」。

## 6. アクセシビリティ（最重要）

> CVD 三重冗長は **MVP 受け入れ条件**（DESIGN.md §2 決定3）。a11y はオプションでなく完成条件。

### 6.1 色覚多様性（CVD）三重冗長 — 最優先

- **色＋アイコン＋ラベルの3経路で種別を伝える**（cardMeta.ts の設計どおり）。どれか1つが欠けても種別が分かること。
  - 例: ⚡＋「イベント」＋オレンジ。色が区別できなくても「⚡」と「イベント」で確定。
- 検証: 1型/2型/3型色覚・グレースケールでスクショを撮り、**全カード種別が区別可能**であることを受け入れテストにする（プロトタイプ→Playwright スクショ→目視）。
- 状態色も冗長化: 正=緑＋`✓`＋テキスト、誤=赤＋`✗`＋シェイク＋トースト文。`done/current/locked` ノードはアイコン（✅/⚡/🔒）で区別。
- スウォッチ単独で意味を担わせない（図鑑・凡例・リザルトすべて「色＋ラベル」セット）。

### 6.2 コントラスト比（WCAG 2.1 AA）

- 本文テキスト ≥ 4.5:1、大きい文字（≥18.66px bold / ≥24px）≥ 3:1 を満たす。
- カードの ink 値（cardMeta.ts）は各背景色に対しコントラストを確保済み（例: event `#FFB74D` 上に `#3a2600` = 約 8.6:1）。**この ink を勝手に薄くしない**。
- UI地のテキスト: `--ink #2b2b2b` on `--paper #faf8f4` ≈ 13.4:1。`--ink-muted #6b6499` on paper ≈ 5.0:1（本文AAクリア）。`--ink-faint` は装飾専用（本文に使わない）。
- ダーク画面（リザルト/完走）: 白文字 on `--accent-dark #3a2a8c` ≈ 9.4:1。opacity を下げた補助文も .85 までに留め 7:1 を確保。
- フォーカスリングは `--accent` の 3px outline ＋ 2px offset。背景に対し 3:1 以上（紫はオレンジ/白板上で十分）。

### 6.3 キーボード操作

- **すべての機能をキーボードのみで達成可能**にする（D&D 含む）。
- dnd-kit `KeyboardSensor`: カードに focus → `Space/Enter` で掴む → `矢印キー`で移動先スロットへ → `Space/Enter` で置く → `Esc` でキャンセル。`@dnd-kit/accessibility` の `announcements` で操作を読み上げる。
- 代替として §5.3 の 2ステップ（カード Enter → スロット Enter）も全要素 `tabindex` で到達可能に。
- フォーカス順序は論理的に（appbar → 盤 → ヒント → CTA → 手札）。`focus-visible` で明確なリング。
- モーダル/トーストは focus trap しない（status は読み上げのみ）。画面遷移時は新画面の見出しへ focus を移す。

### 6.4 スクリーンリーダー

- カード: `role="button"` ＋ `aria-label="イベント: 注文が確定した"`（種別＋内容）。
- スロット: `aria-label="スロット1、空 / 配置: 注文が確定した"`。状態を読み上げる。
- 弾きトースト・接続結果は `role="status" aria-live="polite"` で通知（プロト踏襲）。
- MODE2 不変条件チェッカーは `aria-live` で「未接続2 → 未接続1 → 充足」を逐次読み上げ（孤立コマンドの解消が耳で分かる）。
- 凡例・図鑑は装飾 `aria-hidden` にせず**読み上げ対象**（用語学習に必須情報のため。プロトの `aria-hidden` 方針を改める）。
- 言語属性 `lang="ja"`、英語用語には `<span lang="en">Command</span>`。

### 6.5 効果音オフでも遊べること

- **音は完全に補助**。すべての音イベントに**視覚等価物**がある（スナップ=着地アニメ、正解=緑パルス＋テキスト、誤=シェイク＋トースト、星=拡大＋★表示）。
- 設定で音 ON/OFF 切替（既定 OFF も検討＝公共の場・既定で鳴らさない安全側）。OFF でも★3 到達・完走に一切支障なし。
- これは受け入れ条件: **音を切ったプレイスルーで全ステージ★3が取れる**ことを検証する。

### 6.6 モーション配慮

- `@media (prefers-reduced-motion: reduce)`: シェイク・パルス・ノード鼓動・コンフェッティ・overshoot を無効化（`animation: none` / `transition-duration: 0.01ms`）。配置自体は瞬時に成立させる（機能は失わない）。
- 自動再生する動きは「current ノードの鼓動」のみ。reduced-motion で停止。

### 6.7 その他

- タップターゲット ≥ 44×44px（付箋 min 88×64、ボタン padding 14px で担保）。
- `viewport-fit=cover` ＋ `env(safe-area-inset-*)` でノッチ/ホームバー回避。
- ズーム禁止しない（`user-scalable` を no にしない）。`-webkit-text-size-adjust:100%`。
- 文字サイズはユーザー設定（rem ベース）に追従。固定 px をレイアウト幅以外で多用しない。

## 8. 実装チームへの申し送り（最優先3つ）

### 申し送り1: CVD 三重冗長を「後付け」にしない（最優先・MVP受け入れ条件）
色＋アイコン＋ラベルの三重冗長は、コンポーネントの**初版から組み込む**こと。`Sticky` コンポーネントは `kind` を受け取り `CARD_META[kind]` から color/ink/icon/labelJa を**必ず3つとも描画**する API にし、「アイコンだけ」「色だけ」のバリアントを作れない構造にする。これは a11y であると同時に EventStorming 学習効果そのもの。最後にアイコンを足す進め方は禁止（必ず漏れる）。**グレースケール＆色覚シミュレーションでの区別可能性を MVP のリリース判定に入れる**。

### 申し送り2: juice は「②学習の誠実さ」を一度も侵さない範囲で（優先順位の死守）
演出は気持ちよさのためにあるが、`DESIGN.md` の優先順位 **①正確さ > ②誠実さ > ③juice** を破った瞬間に企画が転落する。具体的には——(a) 半順序判定で「唯一の並びだけ正解」に見せる演出をしない（順序は議論の対象という誠実さを守る）、(b) 弾きは不快な大音量・過剰シェイクで罰しない（失敗の心理的コストを上げない）、(c) 星・ポイントを射幸的に盛らない（過正当化効果）。**演出を足すときは「これは学びを助けるか、快楽報酬を作っているだけか」を毎回自問**。

### 申し送り3: モバイル D&D とタップ代替を「同時に」初日から（離脱L2の直撃回避）
第一ターゲットはスマホで触る独学者。dnd-kit の touch/keyboard sensor とタップ2ステップ配置を**最初のスプリントで両方**動かすこと（どちらか後回しは不可）。最大の離脱点はここ。**実機（iOS Safari / Android Chrome）でドラッグ・タップ・キーボードの3経路すべてで Stage1 が完走できる**ことを、見た目を整える前に通す。`touch-action: none` の付け忘れ・`DragOverlay` 不使用はスマホで即破綻するので最初に固める。
