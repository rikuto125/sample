# StormQuest 企画・設計ドキュメント

> grill-with-docs（PdM / ドメインエキスパート / 技術アーキ / デザイナーの4視点）で揉んだ結果の確定版。
> 変更するときは「なぜ」を ADR（`docs/adr/`）に残すこと。

## 1. プロダクト定義

**StormQuest** = EventStorming の **記法と語彙を、手を動かして、無料で、非同期に学べる入門ドリル**。

- ワンライナー: **「本を読んでも掴めなかったEventStormingの"色と文法"を、15分で手が覚える。」**
- ポジショニング: **Flexbox Froggy of EventStorming**
- 第一ターゲット: **独学エンジニア（本/動画で記法に挫折した層）**。研修・ワークショップ用途は副次的。

### 正直な境界（誇大広告しない）

EventStorming の最大の学びは「協働で認識を合わせる対話」だが、1人用ゲームではそこは扱えない。
本プロダクトは **「記法と語彙の習得に限定した入門」** と位置づけ、完走画面で「次は実ワークショップで人と認識を合わせる体験を」と明示的に接続する。

## 2. 最重要の意思決定3つ（後戻り不可。死守）

### 決定1: 「ちょうど1つのトリガー」を捨て、「少なくとも1つ＋因果の連鎖」を教える
EventStorming の肝は **Event → Policy → Command → Event の因果連鎖**。
- Phase 8 の表現を **「孤立コマンド（起動経路の宙吊り）をゼロにする」** に置き換える。
- 「ちょうど1つ」はステージ内ルールとしては可。ただし解説で「実際は複数経路ありうる」と必ず補足。
- **最終ステージで Event→Policy→Command の連鎖を1本だけ組ませる**（これを入れないと MODE2 は色当てクイズに堕する）。

### 決定2: 採点を「唯一の正解配置」でなく「半順序制約の充足」にする
EventStorming に唯一解はない。全順序を強制すると「並びには正解がある」という嘘を教える。
- **順序制約の集合（半順序）＋複数正解トリガー**で採点。
- 解説で「現場では粒度も順序も議論になり、それが発見だ」と明言する。

### 決定3: 「4ステージMVP＋計測」で公開し、検証してから拡張する
12ステージ作り込みからの公開が最大のリスク。
- **MVP = MODE1 × 3 + MODE2 × 1 = 4ステージ**。
- dnd-kit でモバイル & a11y を初日から確保。
- CVD（色覚多様性）三重冗長化を MVP 受け入れ条件にする。

## 3. 守るべき優先順位（逆にした瞬間に企画は転落する）

**① ドメインの正確さ > ② 学習の誠実さ（唯一解の嘘をつかない）> ③ ゲームの juice > ④ コンテンツ量**

## 4. EventStorming 記法（業界標準・厳密一致）

色を1つでもズラすと学んだ人が実ワークショップで混乱する。色＋アイコン＋ラベルの三重冗長で表現。

| 種別 | EN | 標準色 | アイコン | 時制/形 |
|---|---|---|---|---|
| ドメインイベント | Domain Event | オレンジ `#FFB74D` | ⚡ | 過去形（〜した/された） |
| コマンド | Command | 水色 `#64B5F6` | ▶ | 現在形（〜する） |
| アクター | Actor | 黄(小) `#FFF176` | 👤 | 人・役割 |
| ポリシー | Policy | 紫 `#CE93D8` | ⚙ | 〜したら、〜する |
| 外部システム | External System | ピンク `#F48FB1` | ☁ | 管理外システム |
| リードモデル | Read Model | 緑 `#A5D6A7` | 📋 | 意思決定のために見る画面 |
| 集約 | Aggregate | 淡黄 `#FFF59D` | 📦 | （第2章以降） |

## 5. MVP スコープ

### 作るもの
1. ゲームエンジン1本（データ駆動・UIから分離した純粋関数）: 半順序判定 / 複数正解トリガー判定 / 採点。
2. MODE1: 3ステージ（正常系 → ダミー混入 → 例外・外部システム混入）。
3. MODE2: 1ステージ（最終で Event→Policy→Command 連鎖を1本）。
4. dnd-kit による D&D（マウス＋タッチ＋キーボード）。
5. 配置の juice（スナップ・効果音・マイクロアニメ・正解FB）。
6. 用語図鑑（コレクション）。
7. realityNote（実務リアリティ解説、初期は短文＋"もっと見る"）。
8. CVD 三重冗長表示（色＋アイコン＋ラベル）と基本 a11y。
9. localStorage 永続化（スキーマバージョン付き）。
10. 完走画面＋シェア用の修了サマリ。

### 作らないもの
- 残りステージ（4ステージの検証が出るまで）/ 第2章コンテンツ / アカウント・サーバー保存 / デイリーストリーク・通知 / 協働モード / UI 多言語化 / レベルエディタ。

## 6. 成功指標（北極星 = 完走率とシェア率。リテンションは追わない）

- MODE1 3ステージ完走率 ≥ 40%
- 完走 → MODE2 着手率 ≥ 50%
- 用語ポップ開封率 ≥ 30%（学習意図の代理指標）

## 7. 技術方針

- React + TypeScript + Vite、GitHub Pages で無料公開。
- 状態管理は Context + useReducer のみ（外部ストア不要）。
- D&D は dnd-kit（touch/keyboard sensor を初日からオン）。
- 判定ロジックは純粋関数 → Vitest で厚くテスト。ステージデータ妥当性検証テストも入れる。
- localStorage はスキーマバージョン付き、壊れたデータは握りつぶしてリセット。
