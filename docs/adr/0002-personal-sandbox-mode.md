# ADR 0002: 個人ワーク（サンドボックス）モードを別系統で追加する

- Status: Accepted
- Date: 2026-06-16

## Context

StormQuest は「お題が決まった学習ステージ（採点あり）」だった。ユーザーから「実務で使うときに自分のドメインイベントを入力して、1人でぽちぽち置くと EventStorming 成果物が完成するツールが欲しい」という要望。

DESIGN.md §5「作らないもの」は **協働モード / レベルエディタ** を除外していた。本機能はその境界に触れるため判断を記録する。

## Decision

**個人ワーク（サンドボックス）モードを、学習ゲームとは完全に別系統で追加する。**

- これは **協働モードではない**（1人用の自由制作）。
- これは **レベルエディタではない**（ゲームの正式ステージデータを編集するのではなく、ユーザーが自分の実務ドメインを分析する成果物を作る）。
- 採点しない（EventStorming に唯一解はない、という DESIGN.md 決定2と整合）。

### 別系統であることの構造的担保
- `engine.ts`（採点）・`progress.ts`（学習進捗）に **import レベルで依存しない**。これを `sandbox.test.ts` の import 境界テストで機械的に保証する（型の構造的部分型では分離を担保できないため、型分離ではなく import 境界で守る）。
- localStorage キーは `stormquest.sandbox`（学習の `stormquest.progress` と別）。
- 色・アイコン・ラベルは必ず `CARD_META[kind]` 由来（ユーザーに色を選ばせる UI を作らない＝記法を歪めない）。Sticky を再利用。

### 成果物の保全
ユーザーの成果物（SandboxWork）は **再生成不能**。星（progress）と違い「version 不一致なら全消し」戦略は採らない。`loadSandbox` は version 不一致でも works を破棄せず保持する。JSON が壊れている場合のみ空に握りつぶす。

### 記法の正確さ（助言・非強制）
時制助言は **event(過去形)/command(現在形)/policy(◯◯したら) の3種のみ**。actor/externalSystem/readModel/aggregate に時制概念は無いので付与しない（捏造しない）。助言であり強制修正はしない（唯一解の嘘をつかない）。

## Consequences

- 学習ゲーム（北極星=完走率/シェア率）は一切変更されない。サンドボックスは別入口・別系統。
- MVP は「カード入力 → 2D 自由配置 → JSON 書き出し」。接続線/ホットスポット/集約マーク/共有URL/画像出力/Supabase 同期は後続（deferred）。
- EventStorming の協働の限界（人と認識を合わせる対話）はサンドボックスでも扱えないため、ハブ/完走と同様に「実ワークショップへ」と明示する。
- 関連: DESIGN.md §5、CLAUDE.md（サンドボックスは採点系の別系統という規約）。
